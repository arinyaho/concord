# Codex adapter — port mapping (stub)

Status: **documented, not implemented.** This directory records how each of the five ports in `plugins/concord/core/ports.js` (`PORT_NAMES = ['lifecycle', 'transcript', 'reviewer', 'command', 'statedir']`) would map onto the Codex CLI, so the seam can be built as a follow-up deliverable without re-deriving the design. No Codex runtime code exists yet — see `GAPS.md` for the blockers that would need resolving first.

Background: `docs/superpowers/specs/2026-07-16-vendor-agnostic-harness-adapter-design.md` defines the port contract and the `adapters/claude-code/` implementation these mappings mirror.

## Port mapping

| Port | Claude Code (built) | Codex CLI mapping (stub) | Status |
| --- | --- | --- | --- |
| `lifecycle` | Reads the `Stop` / `SessionStart` hook payload from stdin (`session_id`, `transcript_path`, `last_assistant_message`) and normalizes it into `NeutralEvent`. | Codex CLI fires its own session-lifecycle hooks with a different event/payload shape than Claude Code's stdin JSON. An adapter would read Codex's native lifecycle payload and normalize it to the same `NeutralEvent` shape core already consumes: `{ sessionId, transcriptPath, lastAssistantMessage?, source }`, with the adapter owning which Codex events map to `stop` (write checkpoint) versus the `startup\|resume\|compact` group (inject). | documented, not implemented |
| `transcript` | Wraps the existing byte-offset delta reader plus Claude Code's message-shape parsing (`e.type === 'assistant'`, `e.message.content`, `tool_use` items) to produce `NeutralEntry[]`. | Codex CLI writes its own transcript format. An adapter would need its own delta reader plus a `mapEntries` function that lifts Codex's turn representation into `NeutralEntry` — `{ role, text, toolCalls }` — the same neutral shape `core/state/extract` already consumes (`text` feeds rationale extraction, `toolCalls` feeds fact extraction). | documented, not implemented |
| `reviewer` | Spawns each review/fix subagent via the `Task` tool (`general-purpose`, clean context); parallel `Task` calls issued in one message run concurrently. | **The crux of this adapter.** Codex CLI has no native parallel, clean-context subagent primitive equivalent to Claude Code's `Task` tool. The mapping is `codex exec "<prompt>"`: a single-shot subprocess invocation that runs a prompt to completion and exits, not an in-session spawn. Each reviewer/fix/gate subagent the driver prose calls for becomes one `codex exec` subprocess run, with the subprocess's stdout/exit used the way a `Task` result is used today. See `GAPS.md` for how a fan-out (multiple subagents at once) degrades under this mapping. | documented, not implemented |
| `command` | `commands/*.md` using `${CLAUDE_PLUGIN_ROOT}` and `$ARGUMENTS`, composed from `core/review-driver.md` + `adapters/claude-code/spawn-include.md`. | Codex CLI has its own prompt/command format (its equivalent of a Claude Code slash command). The neutral `core/review-driver.md` would be composed with a Codex spawn-include in place of `adapters/claude-code/spawn-include.md` — same neutral loop logic, different vendor fragment describing how to invoke a reviewer (`codex exec`, per the `reviewer` row above) and how arguments arrive. | documented, not implemented |
| `statedir` | `resolveStateDirFromTranscript` derives a `state/` sibling of the transcript path (or `resolveStateDirFromCwd` reconstructs it under `CLAUDE_CONFIG_DIR/projects/<slug>/state`); `inject` writes to stdout on `SessionStart`. | A `~/.codex`-rooted, project-scoped state directory (Codex's analogue of `CLAUDE_CONFIG_DIR/projects/<slug>/state`), plus whatever mechanism Codex CLI uses to inject context back into a session — its analogue of Claude Code's `SessionStart` stdout injection. Exact Codex path convention and injection API are unresearched pending the real adapter work. | documented, not implemented |

## Why this is a stub

This is the first-deliverable scope: prove the port seam is real and document where a Codex adapter would attach, without building a Codex runtime, packaging, or tests. Implementing this adapter — actual `lifecycle.js`, `transcript.js`, a `reviewer` spawn strategy, a Codex spawn-include, and `statedir.js` mirroring `adapters/claude-code/` — is the next deliverable, gated on resolving the blockers in `GAPS.md`.
