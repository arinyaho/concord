# Codex adapter — port mapping

Status: **partially implemented.** This directory records how each of the five ports in `plugins/concord/core/ports.js` (`PORT_NAMES = ['lifecycle', 'transcript', 'reviewer', 'command', 'statedir']`) maps onto the Codex CLI. `command`, `reviewer`, and `statedir` are implemented as part of the `review-until-green` deliverable; `lifecycle` and `transcript` remain documented mappings only, deferred alongside session-state/charter work. See `GAPS.md` for resolved decisions (fan-out strategy, install-model choice) and remaining known trade-offs.

Background: `docs/superpowers/specs/2026-07-16-vendor-agnostic-harness-adapter-design.md` defines the port contract and the `adapters/claude-code/` implementation these mappings mirror.

## Port mapping

| Port | Claude Code (built) | Codex CLI mapping | Status |
| --- | --- | --- | --- |
| `lifecycle` | Reads the `Stop` / `SessionStart` hook payload from stdin (`session_id`, `transcript_path`, `last_assistant_message`) and normalizes it into `NeutralEvent`. | Codex CLI fires its own session-lifecycle hooks with a different event/payload shape than Claude Code's stdin JSON. An adapter would read Codex's native lifecycle payload and normalize it to the same `NeutralEvent` shape core already consumes: `{ sessionId, transcriptPath, lastAssistantMessage?, source }`, with the adapter owning which Codex events map to `stop` (write checkpoint) versus the `startup\|resume\|compact` group (inject). | not implemented (deferred) |
| `transcript` | Wraps the existing byte-offset delta reader plus Claude Code's message-shape parsing (`e.type === 'assistant'`, `e.message.content`, `tool_use` items) to produce `NeutralEntry[]`. | Codex CLI writes its own transcript format. An adapter would need its own delta reader plus a `mapEntries` function that lifts Codex's turn representation into `NeutralEntry` — `{ role, text, toolCalls }` — the same neutral shape `core/state/extract` already consumes (`text` feeds rationale extraction, `toolCalls` feeds fact extraction). | not implemented (deferred) |
| `reviewer` | Spawns each review/fix subagent via the `Task` tool (`general-purpose`, clean context); parallel `Task` calls issued in one message run concurrently. | **The crux of this adapter, implemented.** Codex CLI has no native parallel, clean-context subagent primitive equivalent to Claude Code's `Task` tool, so the mapping is `codex exec "<prompt>"`: reviewer/verify/gate subagents that the driver calls for in parallel are launched as concurrent backgrounded `codex exec` subprocesses, each writing its own JSON artifact to the state directory for the main session to collect on exit; the fix subagent is the one exception, run one-at-a-time via `codex exec --cd <repoRoot> --sandbox workspace-write --add-dir <stateDir> --skip-git-repo-check "<prompt>"` since it edits the working tree directly and its artifact lives under `~/.codex`, outside `--cd`. Encoded in `adapters/codex/spawn-include.md`. Validated by composition (the include is written and wired into the command) and by a converged direct-verb end-to-end run against real Codex CLI 0.144.5, including a real fix committed by a `codex exec` subagent (`docs/superpowers/specs/2026-07-16-codex-e2e-note.md`); the interactive slash-command dispatch entrypoint itself remains unconfirmed. See `GAPS.md` for the resolved decision and remaining trade-offs (fan-out process cost, sandbox requirement). | **implemented (review-until-green)** |
| `command` | `commands/*.md` using `${CLAUDE_PLUGIN_ROOT}` and `$ARGUMENTS`, composed from `core/review-driver.md` + `adapters/claude-code/spawn-include.md`. | **Implemented** as a native flat Codex plugin at `plugins/concord-codex/`: `.codex-plugin/plugin.json` (manifest), `commands/review-until-green.md` (the neutral `core/review-driver.md` loop composed with `adapters/codex/spawn-include.md` in place of the Claude Code include), and `bin/review-cli.js` (the review-CLI shim the command invokes as `node "./bin/review-cli.js" <verb> ...`, plugin-relative with an absolute-path fallback noted for the UNCONFIRMED case in `GAPS.md`). Validated by composition and by a converged direct-verb end-to-end run against real Codex CLI 0.144.5 (`docs/superpowers/specs/2026-07-16-codex-e2e-note.md`); the interactive command-dispatch entrypoint — i.e. whether the plugin-relative `<review-cli>` path resolves inside a live, interactively-invoked Codex session — remains unconfirmed. | **implemented (review-until-green)** |
| `statedir` | `resolveStateDirFromTranscript` derives a `state/` sibling of the transcript path (or `resolveStateDirFromCwd` reconstructs it under `CLAUDE_CONFIG_DIR/projects/<slug>/state`); `inject` writes to stdout on `SessionStart`. | **Implemented** in `adapters/codex/statedir.js`, mirroring the Claude Code shape for a `~/.codex`-rooted, project-scoped state directory. Covered by a unit test. | **implemented (review-until-green)** |

## What's implemented vs. deferred

`command`, `reviewer`, and `statedir` shipped as part of the `review-until-green` deliverable: `plugins/concord-codex/` (manifest, command, review-CLI shim), `adapters/codex/spawn-include.md` (the `reviewer` spawn strategy), and `adapters/codex/statedir.js` (unit-tested). `command` and `reviewer` are validated by composition — the neutral driver composed with the Codex spawn-include produces the command that's wired into the plugin — and by an end-to-end run of the direct-verb loop against a real Codex CLI (0.144.5) that reached a terminal `converged: true` decision, with a real `codex exec` reviewer and a real `codex exec` fix committed, state living under `~/.codex` (`docs/superpowers/specs/2026-07-16-codex-e2e-note.md`). What that run could not exercise is the *interactive slash-command dispatch entrypoint* — it drove the review CLI directly rather than through a live, interactively-invoked Codex session — so plugin-relative `<review-cli>` path resolution remains the one UNCONFIRMED item; see `GAPS.md`.

`lifecycle` and `transcript` remain documented mappings only — no `lifecycle.js` or `transcript.js` exist. Both are deferred alongside session-state/charter work, since they depend on Codex's native session-lifecycle and transcript formats, which that work would need to characterize first.
